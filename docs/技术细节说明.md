# 毕业答辩技术细节说明

## 1. 项目定位

本项目是一个面向教学场景的低代码强化学习实验平台原型，目标不是单纯实现某一种强化学习算法，而是将以下教学环节打通：

- 实验配置
- 算法训练
- 结果可视化
- 教师基准评估
- 实验历史管理
- 课堂数据分析
- Markdown 报告导出

## 2. 技术栈

### 2.1 后端

- `FastAPI`：提供 HTTP API 和 WebSocket 接口
- `Pydantic v2`：做实验请求、结果、环境配置和算法参数的结构化校验
- `NumPy`：支撑表格型强化学习算法计算
- `PyTorch`：实现 DQN 和 REINFORCE 中的神经网络
- `SQLite`：保存实验记录和自定义教师基准

后端依赖定义见：

- `backend/pyproject.toml`

### 2.2 前端

- `Vue 3`前端界面框架
- `TypeScript`带类型系统的前端开发语言
- `Vite`前端开发和打包工具

前端依赖定义见：

- `frontend/package.json`

## 3. 项目结构与模块职责

### 3.1 关键目录

- `backend/app/api/`：接口层，负责路由暴露
- `backend/app/schemas/`：数据模型层，负责请求和响应结构定义
- `backend/app/rl/envs/`：强化学习环境实现
- `backend/app/rl/algorithms/`：强化学习算法实现
- `backend/app/services/`：业务编排层，如训练、作业模板、教师基准、分析、报告
- `backend/app/repositories/`：持久化层，负责 SQLite 读写
- `frontend/src/`：前端页面、状态管理、接口调用、可视化组件
- `docs/`：项目说明文档
- `data/experiments.db`：实验记录和自定义基准的默认数据库文件

### 3.2 启动入口

真正的项目入口不是根目录的 `main.py`，而是：

- 后端入口：`backend/app/main.py`
- 前端入口：`frontend/src/main.ts`

```powershell
cd backend
uvicorn app.main:app --reload
```

以及：

```powershell
cd frontend
npm run dev
```

## 4. 后端总体架构

后端采用比较清晰的分层结构。

### 4.1 API 层

`backend/app/api/routes.py` 负责定义所有接口，包括：

- `GET /api/catalog`：环境和算法参数目录
- `GET /api/assignments`：作业模板
- `GET /api/benchmarks`：教师基准
- `POST /api/benchmarks`：创建自定义教师基准
- `PUT /api/benchmarks/{benchmark_id}`：更新自定义教师基准
- `DELETE /api/benchmarks/{benchmark_id}`：删除自定义教师基准
- `GET /api/analytics/classroom`：课堂分析
- `GET /api/experiments/history`：实验历史
- `GET /api/experiments/{run_id}`：查看单次实验结果
- `POST /api/experiments/run`：同步运行实验
- `WS /api/experiments/stream`：流式运行实验
- `POST /api/reports/render`：导出 Markdown 报告

### 4.2 Schema 层

`backend/app/schemas/experiment.py` 是后端数据模型核心，定义了：

- 环境配置结构
- 算法参数结构
- 实验请求结构
- 训练结果结构
- 教师基准结构
- 作业模板结构
- 分析结果结构

关键设计是使用了 **Pydantic 的 discriminator 联合类型**：

- 环境通过 `environment_id` 区分
- 算法通过 `algorithm_id` 区分

这样前端提交请求后，后端可以自动根据标识决定应该按哪种结构解析。

### 4.3 Service 层

`backend/app/services/training_service.py` 是训练主编排层，负责：

- 创建运行编号
- 创建环境实例
- 选择对应算法训练器
- 收集训练产物
- 封装为统一实验结果
- 决定是否持久化保存

除此之外，`services` 下还有：

- `catalog.py`：构建环境和算法目录
- `assignments.py`：构建作业模板
- `benchmarks.py`：构建和管理教师基准
- `analytics.py`：生成课堂分析数据
- `reporting.py`：导出 Markdown 教学报告

### 4.4 Repository 层

`backend/app/repositories/experiment_store.py` 负责所有 SQLite 读写，包括：

- 保存实验结果
- 查询实验历史
- 加载单次实验
- 保存自定义教师基准
- 查询和删除自定义基准

## 5. 一次实验的完整执行链路
### 5.1 前端组装实验请求

前端主页面集中写在 `frontend/src/App.vue`。页面中维护了：

- 当前角色
- 提交人
- 所选环境
- 所选算法
- 环境参数
- 奖励参数
- 训练参数
- 教师基准和作业模板

用户点击运行后，前端会调用：

- `buildEnvironmentConfig()`：构建环境配置
- `buildPayload()`：构建完整实验请求

### 5.2 前端通过 WebSocket 启动流式训练

前端没有简单地发一个普通 POST 然后等待结果，而是调用 `runExperimentStream()` 建立 WebSocket 连接。

这么做的原因是：

- 训练是逐轮进行的
- 每一轮都会产生奖励、成功率、epsilon、TD 误差等信息
- 前端希望实时显示收敛曲线
- 用户希望在训练中途暂停、继续、取消

所以 WebSocket 更适合这种“长连接 + 实时推送”的场景。

### 5.3 后端校验请求

WebSocket 接收到第一条消息后，后端会用 `TypeAdapter(ExperimentRequest)` 做结构校验。

这一步会校验：

- 环境配置是否合法
- 算法参数是否合法
- `environment_id` 和 `env_config.environment_id` 是否一致
- `algorithm_id` 是否匹配对应参数结构
- 网格大小、坐标范围、阈值范围是否合法

### 5.4 TrainingService 创建环境和训练器

后端在 `TrainingService._create_trainer()` 中：

1. 先根据请求创建环境
2. 再根据算法类型创建对应训练器

支持的环境：

- `gridworld`
- `cliffwalking`
- `windygridworld`
- `frozenlake`

支持的算法：

- `q_learning`
- `sarsa`
- `dqn`
- `reinforce`

### 5.5 训练器逐轮训练并推送进度

训练器会在每一轮生成一个 `EpisodeMetric`，其中包括：

- `episode`
- `reward`
- `td_error`
- `epsilon`
- `steps`
- `success`

同时按 `trace_frequency` 周期记录路径轨迹 `PathTrace`，用于前端轨迹展示和回放。

后端将这些数据包装成 `ProgressUpdate`，通过 WebSocket 推送给前端。

### 5.6 训练完成后统一封装结果

训练结束后，`TrainingService._finalize_result()` 会统一封装 `ExperimentResult`，其中包括：

- `run_id`
- `created_at`
- `request`
- `summary`
- `metrics`
- `path_traces`
- `policy_grid`

如果用户勾选了保存结果，则写入 SQLite。

## 6. 环境实现细节

项目中的环境都是离散网格环境，统一遵循相同的接口协议：

- `reset()`
- `state_to_index()`
- `step(action)`
- `cell_token(row, col)`

这意味着算法层不需要知道具体是哪种环境，只需要依赖统一接口即可。

### 6.1 状态编码方式

所有环境都采用统一的状态索引编码：

```text
state_index = row * cols + col
```

这样做的好处是：

- 表格型算法可以直接用二维 `Q[state, action]`
- DQN 和 REINFORCE 也可以把离散状态转为 one-hot 向量

### 6.2 GridWorld

`GridWorld` 是基础教学环境，包含：

- 起点
- 终点
- 障碍
- 陷阱

特点：

- 撞墙或撞障碍不会移动
- 到达陷阱或终点会终止回合

它适合讲解：

- 值函数更新
- 奖励设计
- 探索率衰减
- 收敛轨迹

### 6.3 CliffWalking

`CliffWalking` 是悬崖行走环境，最大特点是：

- 掉进悬崖不会直接结束回合
- 会给一个很大的负奖励
- 然后把智能体重置到起点

这个环境特别适合比较：

- Q-Learning 的激进路径
- SARSA 的保守路径

### 6.4 WindyGridWorld

`WindyGridWorld` 的关键在于每一列都有风力强度，智能体移动后会被向上吹若干格。

它适合讲：

- 环境动力学变化
- 状态转移的不确定结构
- 策略如何学会“提前补偿”

### 6.5 FrozenLake

`FrozenLake` 增加了两种难点：

- 稀疏奖励
- 滑移概率

滑移的实现方式不是完全随机，而是在一定概率下偏向左右方向：

- 原动作不变
- 或向左/向右偏转一个动作索引

这个环境适合解释：

- 随机转移
- 稀疏奖励
- 高方差训练

## 7. 算法实现细节

### 7.1 Q-Learning

Q-Learning 是 **离策略** 方法，核心更新目标是：

```text
Q(s, a) <- Q(s, a) + alpha * (r + gamma * max_a' Q(s', a') - Q(s, a))
```

项目中的实现特点：

- 使用 `epsilon-greedy` 选动作
- `epsilon` 每轮衰减
- `td_error` 记录每轮平均 TD 误差
- 最终策略网格从 `Q` 表中选每个状态的最大动作

> Q-Learning 虽然当前可能执行的是探索动作，但更新时使用的是下一状态的最优动作价值，所以它更偏向学习“理论最优策略”。

### 7.2 SARSA

SARSA 是 **同策略** 方法，核心更新目标是：

```text
Q(s, a) <- Q(s, a) + alpha * (r + gamma * Q(s', a') - Q(s, a))
```

它与 Q-Learning 的关键区别是：

- 下一步更新不是取 `max`
- 而是取“当前策略真正选择的下一动作”

因此它更能反映在线探索时的真实风险。

> 因为 SARSA 把探索行为本身也纳入更新目标，所以在风险环境里通常更保守。

### 7.3 DQN

DQN 在本项目中主要用于展示函数逼近能力。

实现包含：

- `policy_net`
- `target_net`
- 经验回放池 `replay_buffer`
- 定期参数同步 `target_sync_interval`

状态输入不是图像，而是 **one-hot 编码后的离散状态向量**。

训练流程是：

1. 与环境交互得到转移样本
2. 放入经验回放池
3. 样本足够后随机采样 batch
4. 用 `policy_net` 计算当前 Q
5. 用 `target_net` 计算目标 Q
6. 通过 MSE 损失反向传播
7. 按间隔同步目标网络

为什么需要 target net？

> 如果用同一个网络同时生成当前值和目标值，训练目标会不断漂移，容易造成不稳定。目标网络可以提供相对稳定的监督目标。

### 7.4 REINFORCE

REINFORCE 是最基础的策略梯度方法，它不是学习 Q 表，而是直接学习策略网络。

核心思想：

1. 跑完整回合
2. 收集每一步的 `log_prob`
3. 计算每一步的折扣回报 `return`
4. 用 `-log_prob * return` 更新策略

本项目还做了一个处理：

- 对回报做标准化，降低训练方差

需要注意一个项目内的实现细节：

- 统一的指标结构里保留了 `epsilon` 字段
- 对 REINFORCE 来说，这里并不表示 epsilon
- 实际存放的是归一化后的策略熵分数

这是为了复用统一前端图表结构。

## 8. 流式训练与训练控制

这是项目比较有展示价值的一部分。

### 8.1 为什么要流式训练

如果使用普通同步接口，前端只能在训练结束后一次性拿到结果，体验会很差，也不适合教学。

流式训练的优势：

- 学生可以实时看到奖励变化
- 教师可以现场展示收敛过程
- 能支持训练暂停、恢复、取消
- 更像实验平台，而不是单次离线脚本

### 8.2 暂停、继续、取消如何实现

后端通过 `StreamController` 控制训练状态，内部核心机制是：

- `asyncio.Event`
- `checkpoint()`

训练器在每个 episode 开始前和每一步执行前都会调用 `checkpoint()`：

- 如果处于暂停状态，就等待
- 如果已经取消，就抛出 `TrainingCancelledError`

这意味着按钮不是前端假交互，而是真正作用到了训练循环内部。

## 9. 前端页面逻辑与可视化实现

### 9.1 前端主页面设计

前端大部分交互逻辑集中在 `frontend/src/App.vue` 中，包含：

- 实验配置区
- 教学作业区
- 教师基准区
- 结果面板
- 策略视图
- 轨迹样本
- 最近实验
- 教师分析
- 轨迹回放
- 实验对比

这实际上是一个“单页面实验工作台”。

### 9.2 为什么前端用一个大页面

因为当前项目是毕业设计原型，核心目标是快速完整展示教学流程，而不是做复杂的路由系统。

优点：

- 所有状态在一个页面中可统一管理
- 交互链路短
- 便于现场答辩演示

缺点：

- `App.vue` 文件较大
- 后续如果工程化升级，应拆分 composables 和子页面

### 9.3 图表实现方式

项目没有引入 ECharts 或 Chart.js，而是自己用 SVG 做了简化折线图。

组件包括：

- `MetricChart.vue`
- `ComparisonChart.vue`

实现方式：

1. 先取所有点的 `min/max`
2. 做归一化映射到 SVG 坐标
3. 通过 `polyline` 渲染折线

这个设计的优点是：

- 依赖少
- 容易控制
- 足够支撑毕业设计展示

### 9.4 策略网格与轨迹回放

`PolicyGrid.vue` 用于显示每个网格单元的最终策略动作：

- `U/R/D/L`
- `START/GOAL/BLOCK/TRAP/CLIFF/HOLE`

`TraceReplayGrid.vue` 负责逐步回放采样路径，支持：

- 当前步高亮
- 已访问路径标记
- WindyGridWorld 风力列附加标记

> 不仅能展示最终指标，还展示了智能体走过的路径，这样能把强化学习的“行为过程”可视化出来。

## 10. 作业模板与教师基准

### 10.1 作业模板

作业模板是面向学生的教学入口，内容包括：

- 作业标题
- 作业简介
- 学习目标
- 推荐环境与算法参数
- 关联教师基准

学生一键加载作业模板后，不需要手写实验代码，只需要在预设基础上调整参数即可。

### 10.2 教师基准

教师基准是平台中很关键的教学评价机制。

一个基准包括：

- 标准实验配置
- 指标阈值
- 教师说明

系统判断实验是否达标时，不是只比较成功率，而是先检查：

- 算法一致性
- 环境配置一致性
- 训练轮次是否达到要求

之后再逐项比较：

- 平均奖励
- 最佳奖励
- 成功率
- 稳定窗口成功率

这部分体现了项目不是“只会训练”，而是加入了“教学评价逻辑”。

### 10.3 自定义教师基准

系统支持教师创建、更新、删除自定义基准。

权限控制方式是轻量级的：

- 请求头中携带 `X-Actor-Role`
- 只有 `teacher` 才允许增删改基准

这不是完整认证系统，但足以支撑毕业设计原型中的角色区分。

## 11. 课堂分析与历史实验

### 11.1 实验历史

平台会把实验结果持久化保存，历史记录中保留：

- 运行编号
- 提交者
- 角色
- 作业归属
- 环境
- 算法
- 平均奖励
- 成功率

这样前端可以做：

- 历史列表展示
- 单次结果复现
- 多次实验对比

### 11.2 课堂分析

`analytics.py` 会对已保存实验做聚合统计，输出：

- 实验总次数
- 学生实验次数
- 教师实验次数
- 平均成功率
- 算法分布
- 环境分布
- 作业分布
- 学生活跃情况
- 基准通过率

这个功能说明平台不仅能服务单个学生实验，还能支持班级层面的教学分析。

## 12. 报告导出设计

平台支持导出 Markdown 实验报告，而不是直接导出 PDF。

这么设计的原因是：

- Markdown 结构简单
- 容易从结构化结果直接生成
- 便于复制到论文附录
- 后续也可继续转 HTML 或 PDF

导出的报告包括：

- 基本信息
- 实验配置
- 训练摘要
- 采样轨迹
- 教师基准评估
- 结果解读

## 13. 数据持久化设计

默认数据库文件是：

- `data/experiments.db`

持久化层不是只保存少量摘要字段，还会保存整个实验结果的 JSON `payload`。

这样做的好处是：

- 后续可以完整恢复实验
- 能支持历史回放
- 能支持对比分析
- 便于模型升级时兼容旧数据

另外，仓储层还包含一个兼容逻辑：

- 如果发现旧版 JSON 结果文件，会尝试导入到 SQLite

这体现了系统对数据迁移的考虑。

## 15. 一些框架和依赖的选择说明

### 问题 1：为什么要做低代码，而不是直接写强化学习脚本？

> 因为教学场景关注的是实验设计、参数理解和结果分析，不希望学生一开始就被大量样板代码阻碍。这个平台把环境配置、算法参数、训练轮次等做成表单，降低了实验门槛。

### 问题 2：为什么用 WebSocket？

> 因为训练是逐轮产生数据的，WebSocket 能实时推送奖励、成功率和轨迹，还支持暂停、继续和取消，更适合交互式实验平台。

### 问题 3：Q-Learning 和 SARSA 的区别是什么？

> Q-Learning 是离策略，更新时看的是下一状态的最优动作价值，所以更偏向理论最优；SARSA 是同策略，更新时看的是当前策略真正会执行的下一动作，因此更能反映探索风险，在高风险环境中通常更保守。

### 问题 4：为什么 DQN 还要用经验回放和目标网络？

> 经验回放打破样本相关性，提高样本利用率；目标网络让训练目标相对稳定，减少训练震荡。

### 问题 5：为什么 SQLite 就够了？

> 这是毕业设计原型，SQLite 的部署成本最低，不需要额外数据库服务，而且足够支持实验历史、基准、课堂分析这些核心功能。

### 问题 6：这个项目有没有完整登录系统？

> 当前没有接入完整认证系统，只做了轻量级角色元数据和教师接口限制。这是有意的范围控制，目的是先把实验编排、教学分析和基准评估主线做完整。

### 问题 7：为什么前端主页面写得比较集中？

> 这是为了服务原型展示效率，把教学实验工作台放在同一页面中，便于演示完整流程。后续如果继续工程化，可以再拆分状态和页面模块。

### 问题 8：教师基准为什么不能只比较成功率？

> 因为环境规模、奖励设计、障碍分布和训练轮次都会影响任务难度。如果这些前提不一致，只比较成功率没有公平性，所以系统先检查配置一致性，再比较指标阈值。

## 16. 当前已知限制

## 18. 答辩时的最短概括版本

> 项目采用前后端分离设计。前端用 Vue 3 和 TypeScript 提供低代码实验配置界面，后端用 FastAPI 暴露 HTTP 和 WebSocket 接口。实验请求经过 Pydantic 强类型校验后，由 TrainingService 统一调度环境、算法和持久化。算法层实现了 Q-Learning、SARSA、DQN 和 REINFORCE，环境层实现了 GridWorld、CliffWalking、WindyGridWorld 和 FrozenLake。训练过程通过 WebSocket 实时推送指标，支持暂停、继续和取消。实验结果保存到 SQLite，并进一步支撑教师基准评估、历史对比、轨迹回放、课堂分析和 Markdown 报告导出。
