# 毕业设计项目说明（答辩优化版）

## 1. 课题名称

低代码强化学习教学平台的设计与实现

如果论文题目需要更完整的学术表达，也可表述为：

面向教学场景的低代码强化学习实验平台设计与实现

## 2. 研究背景与意义

强化学习是人工智能课程中的重要内容，但在实际教学中，相关实验通常依赖 Python 编程基础、算法调参经验和较长的结果观察周期。对于初学者而言，常见问题主要包括：

- 实验配置依赖手写代码，入门门槛较高
- 学生更容易陷入语法和工程细节，而不是理解算法思想
- 训练过程缺少直观反馈，不利于观察奖励变化、探索策略和收敛趋势
- 实验结果分散保存，教师难以统一查看学生完成情况和基准达成情况

因此，设计一个面向教学场景的低代码强化学习实验平台具有明确意义。一方面，它可以通过表单化配置和可视化反馈降低实验使用门槛；另一方面，它能够把实验执行、结果保存、教师基准和课堂分析整合到一个系统中，为强化学习课程教学提供更好的支撑。

本课题的价值不在于提出新的强化学习算法，而在于围绕教学流程，完成一套可配置、可交互、可分析的实验平台原型，实现“算法教学平台化”的目标。

## 3. 研究目标与主要内容

### 3.1 研究目标

本课题拟构建一个面向离散强化学习教学任务的低代码实验平台，使教师和学生能够在无需手写训练脚本的前提下完成实验配置、训练执行、过程观察、结果评估和实验归档。

### 3.2 主要研究内容

结合当前项目实现，课题主要围绕以下内容展开：

1. 设计低代码实验配置机制，将环境参数、奖励函数、算法超参数和训练轮次组织为统一表单。
2. 构建强化学习训练后端，支持典型离散环境与多种代表性算法。
3. 设计流式训练机制，实现训练过程实时反馈以及暂停、继续、取消控制。
4. 实现实验结果持久化、历史记录查询、轨迹回放和多次实验对比。
5. 面向教学场景补充教师基准、作业模板、课堂分析和实验报告导出能力。

## 4. 系统需求分析

### 4.1 功能需求

从教学使用角度出发，系统应满足以下功能需求：

- 支持学生或教师以低代码方式完成实验配置
- 支持多个经典离散环境的训练任务
- 支持多种强化学习算法切换与参数设置
- 支持实时展示奖励值、探索率、误差等训练指标
- 支持实验暂停、继续与取消
- 支持实验结果保存、历史查询与结果复现
- 支持教师设置基准，并对学生实验进行达标评估
- 支持教师查看班级实验汇总、作业分布和学生活跃情况
- 支持导出结构化实验报告

### 4.2 非功能需求

作为毕业设计原型系统，项目还需满足以下非功能要求：

- 易用性：界面应尽量降低强化学习实验操作门槛
- 可扩展性：环境和算法模块应支持后续新增
- 可维护性：前后端分层清晰，便于继续迭代
- 轻量化：部署与运行成本低，适合课程演示和论文验证

## 5. 总体方案与系统架构

### 5.1 总体方案

项目采用前后端分离架构。前端负责实验配置、交互展示和结果可视化；后端负责环境构建、算法训练、数据持久化和分析服务；数据库负责保存实验结果和教师自定义基准。

整体流程如下：

1. 用户在前端完成实验参数配置。
2. 前端通过 HTTP 或 WebSocket 将实验请求发送到后端。
3. 后端根据请求动态创建环境和算法训练器。
4. 训练过程中持续返回指标、轨迹和控制状态。
5. 训练结束后可按需保存实验结果，并支持历史查询、基准评估和报告导出。

### 5.2 系统分层结构

#### 1. 表现层

表现层位于 `frontend/` 目录，基于 Vue 3 + TypeScript + Vite 实现，负责：

- 实验配置界面
- 环境与算法切换
- 实时图表展示
- 策略网格展示
- 轨迹回放与实验对比
- 教师分析面板

#### 2. 服务层

服务层位于 `backend/app/api/` 与 `backend/app/services/`，基于 FastAPI 提供接口，负责：

- 接收实验请求
- 分别返回系统目录信息、内置作业模板目录和教师基准目录
- 编排训练流程
- 生成课堂分析与报告内容

#### 3. 算法与环境层

位于 `backend/app/rl/`，负责：

- 定义教学环境
- 执行强化学习训练
- 输出训练指标、路径轨迹和策略结果

#### 4. 持久化层

位于 `backend/app/repositories/`，当前使用 SQLite 保存实验结果与教师自定义基准，默认数据库文件为 `data/experiments.db`。

### 5.3 关键技术路线

从当前代码实现出发，系统采用了“统一请求模型 + 动态环境创建 + 动态训练器分发 + 流式结果推送 + 结构化持久化”的技术路线。

其核心思想不是为每一种实验单独开发一套页面和后端接口，而是将实验抽象为统一的数据结构，再由后端根据配置动态完成训练编排。这样做有两个直接好处：

- 前端可以用统一表单覆盖多环境、多算法实验
- 后端可以在不改变接口契约的前提下继续扩展环境和算法

从实现上看，这条技术路线大致分为以下几个环节：

1. 前端将实验名称、环境配置、算法配置、训练配置、角色信息等内容组织为统一实验请求。
2. 后端通过 Pydantic 模型对请求进行校验，保证数据结构完整、字段类型正确。
3. 训练服务根据 `environment_id` 创建环境实例，根据 `algorithm_id` 选择对应训练器。
4. 同步模式下直接返回完整实验结果，流式模式下通过 WebSocket 按轮次推送指标。
5. 训练完成后生成统一结果对象，并按需保存到 SQLite。
6. 前端基于统一结果对象渲染图表、策略网格、轨迹回放、历史对比和教师分析。

### 5.4 训练数据流与控制流

系统内部的数据流可以概括为“配置输入 -> 训练执行 -> 指标输出 -> 结果保存 -> 教学分析”。

#### 1. 配置输入阶段

前端将实验配置拆分为若干维度：

- 环境维度
- 奖励维度
- 算法维度
- 训练维度
- 教学维度

其中“教学维度”主要包括提交人、角色、作业信息和教师基准选择，这也是项目区别于普通算法实验脚本的重要地方。

#### 2. 训练执行阶段

后端训练服务先为实验生成唯一 `run_id` 和创建时间，然后构建环境对象与算法训练器，最后启动训练过程。对于流式训练，还会注册进度回调函数，用于持续接收训练过程中的统计信息。

#### 3. 指标输出阶段

训练过程中会持续产生以下数据：

- 每轮奖励
- `epsilon` 维度指标：对 Q-Learning、SARSA 与 DQN 表示探索率，对 REINFORCE 表示归一化策略熵
- `td_error` 维度指标：对 Q-Learning 与 SARSA 表示平均 TD 误差，对 DQN 表示平均损失，对 REINFORCE 表示策略损失
- 成功状态
- 运行平均奖励
- 运行成功率
- 采样路径轨迹

这些数据既用于前端实时渲染，也会在实验完成后成为持久化结果的一部分。

#### 4. 结果保存阶段

训练结束后，后端会将原始实验请求、训练摘要、过程指标、采样轨迹和策略网格统一封装为结果对象，并在用户选择保存时写入数据库。

#### 5. 教学分析阶段

一旦结果被结构化保存，系统即可进一步完成：

- 历史实验查询
- 实验详情加载
- 多次实验对比
- 教师基准评估
- 班级实验分析
- Markdown 报告导出

## 6. 关键模块设计与实现

### 6.1 低代码实验配置模块

该模块通过表单取代手写实验脚本，支持统一配置以下内容：

- 实验名称
- 提交人和角色信息
- 环境类型与环境参数
- 奖励参数
- 算法类型
- 算法超参数
- 训练轮次、随机种子和轨迹采样频率
- 结果是否持久化

这部分是项目“低代码”特性的核心体现。

从技术实现角度看，低代码并不意味着“没有代码”，而是将原本分散在实验脚本中的关键控制变量，前移并抽象为统一可填写的数据结构。这样，用户面对的是配置项，而不是训练脚本。

该设计带来的直接收益包括：

- 减少学生在代码模板、函数调用和参数拼写上的负担
- 让实验差异集中体现在配置变化上，便于教学对比
- 为后续的结果持久化、基准评估和课堂分析提供统一输入格式

### 6.2 强化学习环境模块

根据当前代码实现，系统已支持以下离散环境：

- GridWorld
- CliffWalking
- WindyGridWorld
- FrozenLake

环境模块采用独立封装方式，便于后续继续扩展迷宫、导航或其他课堂任务。

从实现方式看，环境创建并不是写死在单个训练脚本中的，而是由环境工厂统一负责分发。前端只需要传入环境标识和对应参数，后端即可完成环境实例化。这种设计避免了“前端每增加一个环境就必须新增一套训练接口”的问题。

当前环境支持的典型参数包括：

- 网格大小、起点和终点
- 障碍单元与陷阱单元
- 悬崖区域
- 风力列强度
- 滑移概率
- 步进惩罚、目标奖励、墙体惩罚等奖励项

从教学意义上看，这些参数恰好覆盖了强化学习入门实验中最值得观察的变量，学生可以直观看到环境结构和奖励设计对训练结果的影响。

### 6.3 算法训练模块

当前已实现以下算法训练器：

- Q-Learning
- SARSA
- DQN
- REINFORCE

后端训练服务会根据 `algorithm_id` 动态选择对应训练器，实现统一的请求入口和差异化的算法处理逻辑。

从训练服务实现看，该模块承担了整个后端的编排职责，主要包括：

- 生成实验唯一编号 `run_id`
- 创建环境实例
- 按算法标识选择训练器
- 调用同步训练或流式训练方法
- 汇总训练结果摘要
- 根据配置决定是否持久化

当前训练结果并不是仅返回最终奖励，而是返回统一结构化结果，主要包含：

- 实验基本信息
- 训练摘要
- 每轮指标序列
- 路径采样轨迹
- 最终策略网格

其中训练摘要至少包括：

- 平均奖励
- 最佳奖励
- 成功率
- 稳定成功率

这种结果建模方式对教学平台非常关键，因为它让“训练过程”而不只是“训练结果”成为可分析对象。

### 6.4 流式训练与控制模块

系统通过 WebSocket 提供实时训练功能。训练过程中可持续返回：

- 当前轮次与总轮次
- 单轮奖励
- `epsilon` 维度指标
- `td_error` 维度指标
- 当前平均奖励
- 当前成功率
- 最新采样轨迹

同时支持以下控制操作：

- 暂停训练
- 恢复训练
- 取消训练

这使平台不仅能“提交实验”，还能“观察实验进行过程”。

从技术实现上看，流式训练接口采用事件驱动方式，前后端围绕统一事件类型进行通信。后端会按训练状态推送不同事件，典型事件包括：

- `started`：训练已启动，返回运行编号与创建时间
- `metric`：返回当前轮次训练指标和最新轨迹
- `control`：返回暂停、恢复或取消中的状态反馈
- `completed`：训练完成，返回完整结果
- `cancelled`：训练被用户取消
- `error`：请求校验错误或运行时异常

为了支持暂停、恢复和取消，后端训练流引入了独立控制器对象，由 WebSocket 控制消息驱动训练状态变化。这样，控制逻辑与算法主体训练逻辑被解耦，提升了实现清晰度。

与常见的“训练完一次性返回结果”相比，这种实现更适合教学场景，原因在于：

- 学生可以实时看到参数是否朝正确方向收敛
- 教师可以在演示时即时解释奖励波动和探索行为
- 长轮次训练过程中用户不必被动等待结果

### 6.5 结果可视化模块

前端当前已实现以下可视化能力：

- 奖励曲线
- 探索率曲线
- 误差曲线
- 学得策略网格
- 最新采样路径展示
- 逐步轨迹回放
- 历史实验多次对比

这些功能有助于学生从结果层面理解参数变化、策略改进和训练稳定性。

需要说明的是，前端当前统一维护 `reward`、`epsilon`、`td_error` 三组曲线序列。这样做可以复用同一套图表组件，但不同算法下两项指标的语义并不完全相同：值函数类算法主要展示探索率与 TD 误差，DQN 的 `td_error` 实际对应平均损失，REINFORCE 的 `epsilon` 与 `td_error` 则分别对应归一化策略熵和策略损失。

从前端实现结构看，可视化并不是单一图表，而是围绕不同教学目标拆分成多个独立展示单元：

- 指标图表用于观察训练趋势
- 策略网格用于观察最终决策分布
- 轨迹回放用于观察单次路径行为
- 实验对比用于观察不同配置之间的性能差异

这意味着平台在可视化层面同时覆盖了“过程观察”“结果解释”和“对比分析”三个层次，较好对应了课堂教学中的常见提问方式：

- 奖励是否在变好
- 学到的策略长什么样
- 为什么这次结果比上次更好或更差

### 6.6 教学辅助模块

平台当前已实现若干面向教学组织的功能，而不只是单一训练界面：

- 内置作业模板目录
- 教师基准预设
- 教师自定义基准增删改
- 基准达标评估
- 教师视角课堂分析
- Markdown 实验报告导出

这是本项目区别于普通算法实验脚本的重要部分。

其中，作业模板当前由后端以内置目录形式返回，不单独写入 SQLite；教师自定义基准则会持久化保存，并与内置教师基准一起通过 `/api/benchmarks` 提供给前端使用。

其中，教师基准评估不是简单地比较一个最终分数，而是包含了更细的判断逻辑。根据当前系统设计，基准评估至少涉及以下几个方面：

- 算法是否一致
- 环境配置是否一致
- 训练轮次是否满足基准要求
- 平均奖励、最佳奖励、成功率、稳定成功率等指标是否达标

这种做法的意义在于：系统避免了把不具可比性的实验结果强行放在一起比较，从而提高基准评估的合理性。

课堂分析模块则进一步基于历史保存结果进行聚合，支持从多个维度观察教学情况，例如：

- 按作业维度分析实验完成情况
- 按环境维度分析实验分布
- 按算法维度分析实验分布
- 按学生维度分析活跃度和达标情况

当前课堂分析接口以已保存实验结果为输入，支持通过 `limit`、`assignment_id` 与 `benchmark_id` 控制统计范围；接口返回的是聚合后的课堂统计结果，而不是单条实验轨迹或报告正文。

这使平台从“单次实验工具”进一步扩展为“教学管理辅助工具”。

### 6.7 数据持久化模块

系统通过 SQLite 保存实验结果，并为教师自定义基准提供持久化支持。保存内容包括：

- 实验基本信息
- 提交人和角色信息
- 作业信息
- 环境信息
- 算法信息
- 训练结果摘要
- 完整实验载荷

该设计使后续的历史查询、实验对比、课堂分析和报告导出都有统一的数据基础。

从数据组织方式看，系统当前至少维护了两类核心记录：

#### 1. 实验记录

实验记录包含：

- `run_id`
- 创建时间
- 实验名称
- 提交人
- 角色
- 作业信息
- 环境标识
- 算法标识
- 平均奖励
- 成功率
- 完整 JSON 载荷

这种“关键索引字段 + 完整载荷”的设计兼顾了查询效率与结构灵活性。一方面，教师分析可以直接基于关键字段聚合；另一方面，完整载荷又保留了后续回放和扩展分析所需的信息。

#### 2. 教师自定义基准记录

教师自定义基准记录包含：

- 基准编号
- 创建时间与更新时间
- 基准名称
- 创建者
- 环境标识
- 算法标识
- 完整基准载荷

内置教师基准并不写入该表，而是由后端服务在运行时构造，再与数据库中的自定义基准合并后一起返回给前端。

此外，系统启动时还会尝试导入旧版 JSON 实验结果，这说明当前持久化设计不仅考虑了当前功能，也考虑了历史数据兼容问题。

## 7. 当前已完成的工作与阶段性成果

从当前代码实现情况看，项目已经完成了从“实验配置”到“结果分析”的基本闭环，主要阶段性成果如下：

### 7.1 已完成的功能

- 已完成前后端分离的系统原型搭建

## 8. 项目特色与可阐述创新点

在答辩中，建议将本项目的“创新点”表述为“面向教学场景的系统设计创新”或“教学应用创新”，而不是“提出新的强化学习算法”。这样更符合当前项目的真实定位，也更容易在答辩中成立。

换句话说，本项目的创新不在算法理论层，而在于如何把强化学习实验过程转化为一个可配置、可观察、可评估、可复用的教学平台。当前较为稳妥且可以落到代码实现上的创新点如下：

### 8.1 低代码化的强化学习实验组织方式

将原本需要编写训练脚本的实验过程转化为可配置、可提交、可回看的表单化流程，降低了强化学习课程实验的使用门槛。

其技术支撑主要体现在：

- 前端将环境参数、算法参数、训练参数和角色信息统一组织为结构化请求
- 后端使用统一实验请求模型接收配置，而不是为每类实验写一套独立脚本入口
- 环境创建与算法选择通过标识符动态分发，减少前后端耦合

其教学价值主要体现在：

- 学生将注意力从“如何写代码跑起来”转向“为什么参数这样设置”
- 教师更容易组织同一任务下的多组参数对比实验
- 不同实验结果共享统一的数据结构，便于后续课堂分析

答辩时可以将其概括为：

“本课题将强化学习实验从脚本驱动转化为配置驱动，降低了实验门槛，并为教学过程中的结果对比和数据沉淀提供了统一基础。”

### 8.2 教学流程与实验流程的一体化设计

系统并非只关注训练本身，还将内置作业模板、教师基准、基准达标评估、课堂分析和实验报告整合到统一平台中，更贴近真实教学场景。

其技术支撑主要体现在：

- 系统不仅保存训练结果，还保存提交角色、提交人和作业信息
- 作业模板当前以内置目录形式提供，并通过实验请求携带作业来源信息
- 教师自定义基准采用独立数据结构和独立持久化表进行管理，内置基准由后端服务动态构造
- 前端在实验完成后自动执行基准兼容性检查和阈值评估
- 后端支持课堂分析接口，对历史实验按作业、环境、算法和学生维度聚合，并可选结合作业或教师基准范围统计

其教学价值主要体现在：

- 平台能够服务于“布置任务 - 学生实验 - 教师评估 - 结果归档”这一完整教学闭环
- 教师不再只能依赖学生单独提交截图或代码，而可以直接观察平台中的实验结果
- 平台具备一定课堂管理辅助能力，而不仅是一个单机训练器

答辩时可以将其概括为：

“本项目的一个重点创新是把强化学习实验功能与教学管理需求进行整合，使平台既能运行算法，也能支撑作业组织、基准评价和课堂分析。”

### 8.3 面向教学的流式训练交互机制

传统实验脚本往往采用“启动训练 - 等待结束 - 查看结果”的方式，而本项目引入了流式训练机制，使训练过程本身成为可交互、可观察的教学对象。

其技术支撑主要体现在：

- 后端使用 WebSocket 持续推送训练指标和轨迹信息
- 训练流设计了 `started`、`metric`、`control`、`completed`、`cancelled`、`error` 等事件类型
- 控制器对象负责管理暂停、继续和取消操作，避免控制逻辑直接耦合到算法实现内部

其教学价值主要体现在：

- 学生可以实时观察训练收敛趋势，而不是只看最终结果
- 教师可以在课堂演示中及时解释探索率变化和奖励波动
- 训练过程的实时性增强了实验的交互感和可解释性

答辩时可以将其概括为：

“本项目将强化学习训练过程从黑箱式离线执行转化为可实时观察和控制的交互式过程，提升了教学演示效果和实验理解深度。”

### 8.4 面向教学理解的过程可视化设计

通过奖励曲线、探索率曲线、误差曲线、策略网格和轨迹回放等方式，将训练过程和结果状态直观呈现，增强了实验的可解释性。

其技术支撑主要体现在：

- 将训练指标序列结构化保存，供图表组件直接渲染
- 将策略结果抽象为网格数据，用于展示贪心策略分布
- 将路径采样抽象为轨迹对象，用于单步回放和路径对比
- 将多次实验结果统一建模，支持历史结果并排比较

其教学价值主要体现在：

- 奖励曲线回答“训练效果是否改善”
- 策略网格回答“模型最终学到了什么”
- 轨迹回放回答“智能体具体是如何行动的”
- 历史对比回答“不同参数为什么带来不同结果”

答辩时可以将其概括为：

“本项目强调的不只是训练结果展示，而是从趋势、策略、轨迹和对比四个角度提高实验结果的可解释性。”

### 8.5 便于扩展的模块化架构

系统在环境层、算法层、服务层和持久化层之间建立了相对清晰的边界，便于继续接入新的环境、算法和教学分析能力。

其技术支撑主要体现在：

- 环境模块独立封装，并通过工厂模式统一创建
- 算法训练器彼此独立，由训练服务统一调度
- API 层、服务层、持久化层职责分离
- 实验结果采用统一结构建模，降低新增分析功能的改造成本

其工程价值主要体现在：

- 新增环境时主要修改环境定义和工厂分发逻辑
- 新增算法时主要补充训练器和参数结构
- 新增分析功能时可以复用已有持久化结果

答辩时可以将其概括为：

“本项目在工程实现上采用模块化架构，为后续扩展更多环境、算法和教学功能预留了清晰接口。”

### 8.6 创新点的答辩表述建议

如果老师要求用一句话概括项目创新点，建议优先使用下面这种表述：

“本课题的创新主要体现在面向教学场景，将强化学习实验从传统脚本式操作重构为低代码、可流式观察、可基准评估、可历史分析的平台化流程。”

## 9. 当前不足与后续研究计划

虽然项目已经具备较完整的原型能力，但从系统完整性和工程成熟度来看，仍存在以下不足：

- 当前仅使用轻量级角色元数据，尚未实现完整的认证与权限管理
- 作业模板和部分教学内容仍以内置配置为主，教师自主管理能力仍可增强
- 环境主要集中在离散网格类任务，尚未扩展到更复杂的强化学习场景
- 报告导出目前为 Markdown，尚未形成更正式的 PDF 或打印版输出
- 课堂分析维度仍偏基础，尚可继续补充评分导出、跨作业分析等功能

后续计划可从以下方向展开：

1. 补充用户认证、角色权限和实验归属管理能力。
2. 增强教师侧作业、模板和评价标准管理能力。
3. 丰富轨迹回放与结果解释功能，提高教学反馈质量。
4. 扩展更多实验环境与评价方式，提升平台覆盖范围。
5. 完善导出与展示能力，支持更规范的课程报告和答辩材料生成。

## 12. 项目运行方式

### 12.1 启动后端

```powershell
cd backend
python -m pip install -e .
uvicorn app.main:app --reload
```

默认地址为 `http://127.0.0.1:8000`。

### 12.2 启动前端

```powershell
cd frontend
npm install
npm run dev
```

默认地址为 `http://127.0.0.1:5173`。

前端支持通过 `VITE_API_BASE_URL` 配置后端接口地址。
